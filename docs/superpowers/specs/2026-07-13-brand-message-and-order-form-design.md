## 0. この設計書のスコープ

HEBERESTを「へべす・宮崎・日向の魅力を発信する専門メディア兼ECサイト」として長期的に育てるという既存方針を維持したまま、以下2点を1つのブランド体験として設計する。

1. ブランドメッセージ更新（HEBESU→HEBEREST リネーム、サブタイトル変更、二層タグライン構造）
2. サイト内購入フォームの新設（Journal・Maruhe OS連携を見据えた基盤設計）

実装は行わない。この設計への承認後、フェーズごとに実装計画（writing-plans）を作成し、各実装後にご承認を得てから次に進む。

## 2. 全体アーキテクチャ方針（ブランド体験として一体設計）

```
Journal記事（将来） ──┐
                      ├─→ 商品詳細（Products） ─→ 購入フォーム（/order/[productId]） ─→ 完了
ブランドストーリー ────┘                                     │
                                                              ▼
                                                     注文データ生成 → 保存 → 通知 → (将来)Maruhe OS
```

- Hero・BrandStory・Producer・Productsという「知る→興味を持つ→ファンになる」導線は変更しない
- 「購入する」ボタンの遷移先を、外部（メルカリ）だけでなく自社フォームにも開けるようにする（フォームが安定するまでメルカリ導線は残す＝両立）
- 購入フォームのUIトーン（余白・写真・フォント）は既存デザイントークンを踏襲し、「安っぽい通販フォーム」にしない
- 将来Journal記事から個別商品への導線ができた際に、商品詳細→購入フォームへ自然に接続できるようURL設計をしておく（例: `/order/[productId]` は将来 `/journal/[slug]` からもリンクできる独立した入口にする）

## 4. フェーズB設計：購入フォーム新設

### 4-1. ルーティング設計

ご提示いただいた `/order` `/order/confirm` `/order/complete` の3ページ構成と、Next.js App Routerに適した代替案を比較する。

| 観点 | 3ページ構成（ご提示案） | **推奨: 単一ルート＋クライアント側ステップ** |
|---|---|---|
| URL | `/order` → `/order/confirm` → `/order/complete` | `/order/[productId]`（入力〜確認を1URLで完結）→ `/order/complete?orderId=...` |
| 入力内容の引き継ぎ | ページ遷移のたびに状態をどこかに保持する必要がある（Cookie/セッション/クエリ等） | Reactの状態としてブラウザ内に保持。遷移がないため消えない |
| 戻る操作 | ブラウザバックでページが切り替わり、実装によっては入力が消える | 「確認画面」はステップ表示の切り替えなので、戻ってもその場で入力内容が残る |
| 二重送信対策 | 別ページなので送信は`/order/confirm`側で1回、比較的単純 | `useActionState`の`pending`状態＋ボタン無効化＋冪等キーで対応（後述） |
| 実装量 | 3ルート分のページ・状態受け渡しロジックが必要 | 1ルート＋1完了ルートで済み、状態受け渡しの実装が不要 |
| 完了画面の共有・再読み込み | 独立ページなので自然 | 完了時のみ`/order/complete?orderId=...`へ実際に遷移するため同様に自然 |

**推奨理由**：ご要望の「戻る操作で入力内容が不自然に消えない」「安っぽい通販フォームに見せない（=ページ遷移のカクつきがない）」に最も合致し、かつ個人情報をURLやCookieに載せて次ページへ引き継ぐ必要がない（セキュリティ的にも優位）。完了画面だけは実際のページ遷移にすることで、リロード耐性・共有可能なURL・分析（コンバージョン計測）のしやすさを確保する。

ページ構成案：
```
/order/[productId]     … 入力ステップ（画面内タブで「入力」→「確認」を切替）
/order/complete         … 送信完了（?orderId=ORD-XXXXX を付与、注文番号を表示）
```

商品一覧からの導線は「購入する」ボタンに `href="/order/1kg"` のような商品スラッグを割り当てる形にする（メルカリ外部リンクは当面併存、後述4-9）。

### 4-2. レイヤー構成（Maruhe OS接続を見据えた分離）

```
src/
  lib/
    products.ts          … 商品カタログの単一の真実源（ID・容量・単価・在庫可否）※既存Products.tsxから抽出
    orders/
      schema.ts           … Zodスキーマ（フォーム入力の型・バリデーションルール）
      pricing.ts           … 数量・商品IDから金額をサーバー側で再計算する純粋関数
      order.ts             … 注文データ生成（フォーム入力+価格計算 → Order型オブジェクト）
      store.ts             … 保存層（差し替え可能なアダプター。初期実装はDB or Sheets）
      notify.ts            … 通知層（管理者宛・注文者宛。初期実装はResend）
      types.ts             … Order / OrderItem / Customer などの共通型（将来Maruhe OSへそのままJSON送信できる形）
  app/
    order/
      [productId]/page.tsx … フォームUI（Server Component、初期データ取得）
      [productId]/actions.ts … 'use server' の注文送信Server Action（上記lib層を呼ぶだけ）
      complete/page.tsx     … 完了画面
```

**設計原則**：UIコンポーネントは「フォームを表示し、Server Actionを呼ぶ」以上のことをしない。金額計算・保存・通知のロジックは全て`lib/orders/`に閉じ、UIから直接DBやメールAPIを叩かない。これにより：
- 保存先をSheets→DBに変えても`store.ts`の中身を差し替えるだけで済む
- 将来Maruhe OS APIに送る処理を`notify.ts`や`store.ts`に追加するだけで済み、UIは無変更
- 単体テストがUIなしで書ける（`pricing.ts`や`schema.ts`は純粋関数）

### 4-3. 商品カタログの共通化（既存コードへの改善）

現状`Products.tsx`にハードコードされた配列は表示専用で、金額計算・注文処理に使える形になっていない。`lib/products.ts` に以下の形で抽出する。

```ts
export type ProductId = "05kg" | "1kg" | "18kg" | "3kg"; // 現状注文可能な4商品のみ

export interface Product {
  id: ProductId;
  name: string;
  variantLabel: string; // "0.5kg" など
  unitPriceJPY: number; // 税込・送料無料の実数（例: 250000銭ではなく円単位の整数）
  mercariHref: string;  // 既存の外部リンク（併存用）
  imageSrc: StaticImageData;
}

export const PRODUCTS: Record<ProductId, Product> = { ... };
```

`Products.tsx`はこのカタログを`import`して表示するだけに書き換える（表示ロジックは変えない）。コミングスーン4商品（果汁×2・路地へべす・黄色い秋へべす）は当面カタログに含めず、注文フォームの対象外のままとする。

### 4-4. フォーム項目（初期実装の絞り込み）

ご提示いただいた項目一覧から、**産地直送・季節商品の注文フォームとして成立する最小構成**に絞る。「不要な項目まで詰め込まない」というご要望に基づき、以下を初期実装の対象とする。

**注文者情報（必須以外は任意と明記）**
- お名前 *必須*
- フリガナ *必須*（送り仮名の自動変換は行わず手入力、宅配伝票用）
- 郵便番号 *必須*（ハイフンなし7桁でも自動整形、入力後に住所自動補完を検討）
- 都道府県・市区町村・番地建物名 *必須*
- 電話番号 *必須*（配送連絡用）
- メールアドレス *必須*（確認メール送付用）

**商品情報**（フォーム上は選択済み商品を表示するのみ、計算はサーバー側）
- 商品名・容量・数量（1〜9個のセレクト）・単価・小計・送料・合計 を確認ステップで表示

**配送情報**
- 「注文者と配送先が同じ」チェック（初期値ON、OFFで配送先入力欄を表示）*必須の分岐項目*
- 配送先氏名・郵便番号・住所・電話番号（「同じ」がOFFの時のみ）
- 配送希望日・時間帯 *任意*（生鮮品のため「指定なし」を選べるようにする）
- 備考 *任意*

**初期実装で見送る項目**（理由）
- のしの有無・種類・名入れ … 果物の産地直送ではまだ需要が読めないため、備考欄で代替。需要が見えたら追加
- 年齢確認 … 酒類等の年齢確認が必要な商品を扱っていないため不要

**同意項目**
- 個人情報の取り扱いへの同意 *必須*（プライバシーポリシーへのリンク）
- 注文内容（金額・商品）への同意 *必須*
- キャンセル・返品条件への同意 *必須*（生鮮品の返品不可規定など、特定商取引法表記と整合させる）

> **要確認**: 現状サイトに「特定商取引法に基づく表記」「プライバシーポリシー」はフッターにテキストのみで実ページが存在しない（リンク切れ状態）。フォーム公開前にこの2ページの整備が必須になる。

### 4-5. 金額計算とサーバー側再計算

クライアントは表示用に概算金額を出すが、**送信時はサーバー（Server Action）が商品ID・数量から`lib/orders/pricing.ts`を使って金額を必ず再計算**し、フォームから送られてきた金額表示はいかなる場合も信用しない。送料は初期実装では全商品「送料無料」のため固定0円だが、将来送料が商品や地域によって変わる場合に備え、`pricing.ts`に地域係数を後から足せる形にしておく。

### 4-6. 保存先・通知先の比較

| 方式 | 初期構築コスト | 運用のしやすさ | Maruhe OS接続との親和性 | 備考 |
|---|---|---|---|---|
| **メール通知のみ**（Resend） | 最小 | 注文が増えると一覧管理が困難 | 低（構造化データが残らない） | MVPの土台としては必須だが単独では不十分 |
| **Googleスプレッドシート**（Apps Script Web App経由） | 小〜中 | ノーコードで一覧・検索・ステータス管理が可能、非エンジニアでも扱える | 中（後で列をJSONへ変換すれば移行可） | Google Cloudのサービスアカウント認証が不要なApps Script方式を推奨 |
| **データベース**（Vercel Postgres / Neon＋Drizzle等） | 中 | 構造化・検索性◎、注文番号や状態管理が本格的にできる | **高**（テーブル設計がそのままMaruhe OSの受注データモデルの土台になる） | 移行コストが最も低い |
| **外部フォームサービス** | 小 | 手軽だがデータの所有権・形式を自社でコントロールしにくい | 低 | 依存を増やしたくないという方針にも合わない |
| **将来のMaruhe OS API** | - | - | - | 今回は未接続。後述4-8で境界だけ用意する |

**推奨**: **Resend（通知）＋ データベース（Vercel PostgresまたはNeon、Drizzle ORM）を保存の正とする**。理由：
1. 「数年かけて育てるブランド」「Maruhe OSへの将来接続」を最優先事項に挙げておられるため、最初から構造化データとして持つことが結果的に一番手戻りが少ない
2. Vercelとの統合が標準機能としてあり、追加の大規模な依存パッケージを増やさずに済む（Drizzleは軽量）
3. 管理者向けの注文一覧は、まずは簡易な管理画面（`/admin/orders`をBasic認証等で保護）かSupabase/Neonのダッシュボードで代替し、本格的な管理UIはMaruhe OS側に譲る

**代替案**：DB構築のコミットを今は避けたい場合、Googleスプレッドシート＋Apps Scriptで先にMVPを出し、注文数が増えてきたタイミングでDBへ移行する道もある。この場合も`store.ts`のアダプター構造にしておけば移行は容易。→ **どちらを選ぶかはご判断をお願いしたい点として4-11にまとめる**。

### 4-7. 決済の扱い

今回は決済方法を確定しない。フォーム受注の完了時点では「注文を受け付けました。お支払い方法は追ってメールでご案内します」という運用（受注後に銀行振込等を個別案内）を初期仕様とする。

拡張ポイント：
- `Order.paymentStatus: "pending" | "invoiced" | "paid"` のような状態フィールドを注文データ型に最初から持たせておく
- 将来Stripe等を追加する場合、`/order/[productId]`の最終ステップを「注文確定」から「決済へ進む」に差し替えるだけで済むよう、注文データ生成（`order.ts`）と決済処理を最初から別関数に分離しておく

### 4-8. Maruhe OS連携を見据えたデータ構造

```ts
interface Order {
  orderId: string;        // 例: "HBR-20260713-0001"（サイト内で採番）
  createdAt: string;      // ISO8601
  status: "received" | "confirmed" | "cancelled";
  paymentStatus: "pending" | "invoiced" | "paid";
  customer: {
    name: string; nameKana: string;
    email: string; phone: string;
    postalCode: string; prefecture: string; city: string; addressLine: string;
  };
  shipping: {
    sameAsCustomer: boolean;
    name?: string; phone?: string;
    postalCode?: string; prefecture?: string; city?: string; addressLine?: string;
    desiredDate?: string; desiredTimeSlot?: string;
    note?: string;
  };
  items: Array<{
    productId: string; name: string; variantLabel: string;
    unitPriceJPY: number; quantity: number; subtotalJPY: number;
  }>;
  shippingFeeJPY: number;
  totalJPY: number;
  consents: { privacy: boolean; orderContent: boolean; cancellationPolicy: boolean };
  source: "HEBEREST-web"; // 将来複数チャネルが増えた時の識別子
}
```

この型を`lib/orders/types.ts`に置き、保存層・通知層・（将来の）Maruhe OS送信層すべてがこの型を経由する。JSONとしてそのままシリアライズ可能な設計にしてあるため、将来Maruhe OSへWebhookやAPIで送る際もマッピング作業が最小で済む。

### 4-9. 既存メルカリ導線との併存

- `Products.tsx`の「購入する」ボタンは、注文可能4商品について `/order/[productId]` へのリンクに変更
- 各商品カードに小さく「メルカリShopsでも購入いただけます」という補助リンクを残す（ご要望の「フォームが安定するまで削除しない」を満たす。安定後にメルカリ導線を残すか外すかは別途判断）
- comingSoon 4商品（果汁×2・路地へべす・黄色い秋へべす）はボタン非活性のまま変更なし

### 4-10. セキュリティ・スパム対策

| 項目 | 対応方針 |
|---|---|
| サーバー側バリデーション | Zodスキーマで全項目を再検証。クライアント側の`required`属性やJSは信用しない |
| 入力値のサニタイズ | Zodでの型・長さ・形式チェックに加え、保存前にHTMLタグ等を除去（メール本文へのインジェクション対策） |
| 価格改ざん防止 | 4-5の通り、金額は常にサーバーで商品IDから再計算 |
| CSRF対策 | Server Actionsは**Next.js標準でOrigin/Hostヘッダー検証によるCSRF対策が組み込み済み**（POSTのみ受理、送信元オリジンが一致しない場合は自動的に拒否）。追加のCSRFトークン実装は不要 |
| スパム対策 | Vercel **BotID**（見えないCAPTCHA、Kasada提供）をServer Action内で`checkBotId()`により検証。reCAPTCHA等の「目に見えるチャレンジ」よりブランド体験を損なわない |
| レート制限 | `@vercel/firewall`の`checkRateLimit`をServer Action冒頭で使用し、同一IPからの短時間大量送信を弾く |
| 環境変数 | Resend APIキー等は`.env.local`（Vercelでは環境変数機能）で管理し、`NEXT_PUBLIC_`接頭辞は付けない。`lib/orders/`以外のコードから`process.env`に直接アクセスしない |
| ログのPII配慮 | エラーログに氏名・住所・電話番号を出力しない（注文IDのみ記録） |
| エラー画面 | ユーザー向けエラーは「送信に失敗しました。時間をおいて再度お試しください」等の一般文言のみ。スタックトレースや内部情報は表示しない |
| 個人情報保護方針 | フォーム内にプライバシーポリシーへのリンクを設置（4-4の要確認事項の通り実ページ整備が前提） |
| 保存期間・保存場所 | DB採用の場合はVercel/Neonのデータセンター（東京リージョン選択可）。保存期間の方針は運用ルールとして別途整理が必要（要確認事項） |

### 4-11. 二重送信・冪等性

- フォーム送信ボタンは`useActionState`の`pending`状態で自動的に無効化（連打防止）
- クライアントで生成したUUID（冪等キー）をhidden fieldとして持たせ、Server Action側で「同じキーの注文は再作成せず既存の注文番号を返す」処理にする（ネットワーク不安定時の多重送信対策）
- 完了画面遷移後はブラウザバックしてもフォームの`pending`状態が残っているため再送信されない設計にする
